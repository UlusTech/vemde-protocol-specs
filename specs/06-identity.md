# 06 — Identity

## Overview

Vemde identifies peers by UUID. The UUID is generated by the client and sent in `vemde.session.hello`. It is scoped to the session — the same user connecting to two different sessions may have different UUIDs.

Human-readable identity (display name, avatar, email) is contributed by plugins. The core protocol does not define or require any identity metadata beyond the UUID.

## Base Identity

```
peer_id: UUID v4    ← generated by client at connect time
```

That is all the core protocol knows about a peer.

## Plugin-Contributed Identity

Plugins may contribute identity metadata by sending a `vemde.identity.update` message after the session is established. This metadata is attached to the peer's entry in `vemde.session.welcome` and `vemde.session.joined` as the `metadata` map.

Examples of plugin-contributed metadata:

| Plugin | Metadata keys |
|---|---|
| git | `git.name`, `git.email` |
| github | `github.login`, `github.avatar_url` |
| custom | anything |

Clients decide how to display metadata. A client that knows about `github.avatar_url` may show an avatar. A client that does not know about it ignores it.

## Messages

### `vemde.identity.update` — Server → All Clients

Sent when a peer's identity metadata changes (e.g. after a plugin loads and resolves identity).

**Protobuf**
```proto
message IdentityUpdate {
  string peer_id = 1;
  map<string, string> metadata = 2;
}
```

**JSON**
```json
{
  "peerId": "uuid-v4",
  "metadata": {
    "git.name": "Alice",
    "git.email": "alice@example.com"
  }
}
```

## Notes

- There is no authentication in the Vemde protocol. Whether a peer is who they claim to be is an implementation concern.
- The UUID is trust-on-first-connect. If an implementation needs verified identity, it must handle that at the transport or application layer.
- The official Vemde server uses git config (`user.name`, `user.email`) as the default identity source, contributed via the git plugin.
